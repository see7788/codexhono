# Checklist 任务状态


## 触发规则

- 只要满足以下任一条件，必须自动拆 checklist：用户提问包含多个待解决点；预计会修改多个文件；属于批量改动；需要多阶段验证。
- 触发 checklist 后，先列 checklist 再动手；用户同句明确授权继续时，列完立即执行；执行中用户问进度、状态、做到哪了时，必须补充或同步当前 checklist 状态。
- 用户明确要求“先输出 todolist / checklist / 任务清单 / 修改清单”时，第一段必须直接输出 checklist；禁止先输出解释、技能声明、道歉、寒暄或计划外说明。
- 用户要求先列清单、先不要动手或先确认且未同句授权继续时，输出 checklist 后必须等待用户确认；未确认前禁止修改文件、运行实现命令、推进执行或把未执行事项标为 `[x]`。
- 批量修改、跨文件改动、模板源修改、规则模板修改、生成链路修改、真实接线前准备，必须先输出 checklist；用户未明确授权继续时等待确认，用户同句授权继续时列完直接执行；范围改变时先更新 checklist 再按授权状态处理。
- 模板规则复盘类 checklist 必须覆盖：问题行为、通用规则类型、模板位置、重复风险、规则文本、预期约束效果。
- 涉及对象边界、store 切片、类型出口、Hono 聚合、跨项目 API 的复杂重构必须先 todolist，并按对象边界、公共行为、私有实现、验证项拆分。
- 如果用户只要求复盘规则或补丁建议，不得把 checklist 扩展为当前业务项目实现步骤。
- 简单单点问答或单文件小改不强制使用 checklist。

## 状态同步

- checklist 只记录当前任务的关键目标、将修改的文件或阶段、必要验证项。
- 所有实现任务的最后一项必须是“文档与源码对齐”，并作为不可跳过的完成门。修改源码、配置、命令、公开入口、运行流程或目录结构后必须实际检查 README 和相关文档；用户可见行为、启动方式、配置结构、公开接口或项目结构发生变化时，必须在同一任务中同步修改文档，未同步时标记 `[!]` 且禁止宣称完成。
- 只有对照源码改动、文档内容和 `git diff` 后，才能把“文档与源码对齐”标记 `[x]`；确认无需修改文档时也必须记录具体判断依据，禁止用“无需修改”无依据跳过。
- 执行中按需同步 checklist 状态：`[ ]` 未开始，`[~]` 进行中，`[x]` 已完成，`[!]` 被阻塞。
- 只有 checklist 目标项都处理完并完成必要验证后，才使用“完成了”“已处理完”等收工表达；阻塞时明确卡在哪项和下一步需要什么。
- 验证失败时必须区分本次改动相关失败和既有无关失败；相关验证未完成时标记 `[!]`，无关失败不得阻断本次局部结论，但必须在收尾中列出。
- 如果 checklist 已列出且用户指出“没完成”“你停了”“继续”等，视为继续推进未完成项的确认；必须立刻同步状态并继续执行，不能再次停在计划说明。

## 收尾检查

- 每轮工作收尾前必须检查本轮是否有被用户打断、中途暴露、计划中列出但未完成的事项。
- 未完成事项能继续处理就继续处理；不能处理时记录到根目录 TODO.md，并写清阻塞原因、下一步动作和相关文件。
- 用户批评实现虚假、过度工程化、没按规则、没真实验证、没说明阻塞、单点 helper、提前抽象或不符合习惯时，必须先判定是否暴露通用 AI 行为约束缺口。
- 存在通用规则缺口时，输出模板反推块：应补位置、规则层级、具体命令式规则文本、触发场景、与已有规则的重复关系、预期约束效果。
- 模板反推必须定位到 source.ts 的生成源：全局分流和硬边界放入 agentsMd，具体执行细则放入 skills / xxx-styleskill 的对应 section；禁止建议直接修改 .codex 生成产物。
- 如果已有规则覆盖但本轮未执行，标注为“强化已有规则”，说明替换或并入哪条，禁止新增同义规则制造重复。
- 被批评后禁止只解释原因、只道歉、复述用户要求或只给当前代码修复方案；必须给出可合并到 source.ts 的规则文本。
- 如果用户明确禁止分析或修改当前业务项目，当前代码修正动作写“不适用”，只输出模板补丁建议。
- 收尾回复必须标注实现状态：已真实接线并验证、已接线未验证、未接线等待信息、被阻塞；禁止把未验证或未接线内容表述为完成。
- 收尾回复必须提供文档同步证据：写明“文档已修改”及文件路径和对应变化，或写明“文档无需修改”及具体原因；禁止只写“文档与源码已对齐”而不给检查依据。
- 项目存在 docs:check、文档 lint、提交钩子或 CI 文档门禁时必须执行；高风险公开入口变更触发门禁但 README 未变化时，不得绕过检查或宣称完成。
- Hono/web 项目收尾复扫必须检查：web 侧是否只从服务端包导入 `*Api` 类型；`routers/hono/*` 是否只导出 `typeof app`；是否存在从服务端包导入 DTO、Event、State、Store 或对象类型。
- 边界重构收尾必须复扫：`export type`、`Pick/Omit/Partial`、`Store["x"]`、`Record<string, never>`、router 类型转发、单点 helper 和无外部消费者的 export。
- 相关项目必须跑 `tsc --noEmit` 或项目既有 typecheck；无法验证时必须说明是权限、依赖、脚本缺失还是既有错误导致。
- 如果项目已有明确 TODO 文件或任务规范，沿用既有规范；没有时使用根目录 TODO.md，不把未完成事项散落在回复里。
