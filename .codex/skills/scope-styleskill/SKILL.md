# 作用域风格


## 分流规则

- 前端组件结构、页面交互、组件拆分、UI 临时态和样式放置使用「前端作用域」或「前端样式作用域」。
- 后端 route handler、业务对象边界、实例复用、schema/cache 收敛使用「后端作用域」。
- 抽象、复用、文件拆分和最小作用域先看「通用作用域」；跨文件导出和默认导出看「导出边界」。
- 变量、形参和方法命名只引用 variable-style；业务状态流转只引用 zustand-store-style；网络协议只引用 net-style。

## 通用作用域

- 最小作用域、形参最小化：单点逻辑直接读当前作用域；真实复用后，才把差异提升为形参。
- 先实现后抽象：单方法优先 `export default () => {}`，class 优先 `export default class {}`；无实际复用不预定义文件级常量、类型、helper、配置或形参。
- 具体实现前先做真实实现前置检查：确认真实输入、真实配置、真实调用路径、真实副作用和真实验证方式；缺任一关键条件时先阻塞并列缺项，不先写象征实现。
- 真实边界包括真实 API、真实数据库、真实文件、真实进程、真实服务、真实账号权限或用户明确指定的真实依赖；实现没有触达真实边界时，判定为未真实实现。
- 用户要求具体实现时，必须接入真实调用路径、真实配置、真实文件、真实命令或真实接口；禁止用本地变量、内存状态、静态返回、假成功分支、空 catch、默认兜底值、乐观 UI、mock、stub、dummy、示例数据或自造响应冒充完成。
- 缺少真实实现所需信息时暂停实现并列出缺项；涉及服务器、远端服务或账号能力时，缺少 IP、域名、端口、用户名、密码、token、密钥、路径、进程名、协议、表名、字段名或启动命令中的任一必要项，都必须标记为阻塞。
- 缺少真实信息时，只实现不依赖缺项且可验证的部分；依赖缺项的代码保持未实现或显式阻塞，不创建假配置、假返回、假进程控制或假网络调用。
- 禁止用“逻辑跑通”替代真实可用；构建通过、类型通过、单元测试通过、页面状态变化或日志成功只证明代码路径可执行，不能声明真实依赖已验证。
- 只有实际触发真实依赖并观察到真实返回、真实副作用或真实持久化结果，才允许标注已真实接线并验证；否则标注已接线未验证、未接线等待信息或被阻塞。
- 真实复用才提升：同文件内只有跨方法真实复用、归一化或维护不变量时，才定义文件级成员；禁止单点调用 helper 和为工程化预留的僵尸定义。
- 跨文件真实复用时，优先把消费文件整理为 `目录/index.ts|tsx`，并在同目录 `store.ts` 或对象边界收敛公共能力。
- 不要先猜测、不要预判复用：真实重复出现后才在最小共同作用域归一；禁止为了未来可能复用提前抽象。
- 单点调用：只有一个调用点的常量对象、短文案表、helper、一对一 wrapper、套壳文件、re-export 转发文件，都属于虚假抽象，应内联到真实消费处。
- 单点协议类型、单点响应类型、单点 helper、单点 wrapper 必须内联或保持文件私有；禁止为了看起来完整而导出。
- 归一化放置顺序：单点消费内联到当前消费点；同一视图或路由私有内容放在该视图或路由目录；业务状态、业务参数、状态提示、协议字段和多 action 共用逻辑放进已有切片仓库或业务对象；只有跨业务真实复用且没有既有业务边界时才创建新模块。
- 工程化克制：禁止无复用形参，禁止一对一 wrapper/helper；无入口、无调用、无复用时，禁止先铺目录、类型、配置、bridge、compat、adapter；命名使用 variable-style。
- 小类型由真实对象先定义；大类型只在组合点出现并通过继承或交叉组合形成，禁止先定义巨型类型再用 Pick/Omit/索引访问分发。
- 能继承的类型不套娃；能由实际调用点自动推导的类型不手写、不导出。

## 前端作用域

- React 组件只负责渲染状态、绑定交互和触发仓库 action；禁止组件直接承载业务状态流转。
- 普通 Vite React SPA 项目必须使用 react-router-dom，并且必须存在 src/routers.tsx；默认使用 hash 路由，除非确认部署端具备 history fallback 才可使用 BrowserRouter；组件库、Next/Taro 等框架项目或嵌入式页面按项目既有路由体系处理。
- src/routers.tsx 只定义和导出 React Router 路由对象、路由数组、布局挂载和子路由组合；禁止创建页面私有状态、页面私有事件、业务 action、document 副作用、页面样式对象或组件私有样式。
- 无真实复用时直接 `export default createHashRouter([...])`；禁止先定义 `const router` 再导出，禁止导出 `function Routers()`、`<HashRouter>` 包装组件或自定义 pathname 分发组件；视图逻辑进入对应路由目录，业务流转进入 zustand-store-style「前端仓库」。
- React 项目采用 hash 路由时，只能在路由根入口选择一次 HashRouter/createHashRouter 形态；禁止在页面、布局、子组件或多个入口中重复创建 HashRouter、BrowserRouter、RouterProvider 或自定义 router 实例。
- src/index.tsx 或 src/index.ts 只负责 createRoot、全局 Provider、RouterProvider 挂载和必要的全局资源导入；禁止直接承载页面 JSX、路由判断、页面切换逻辑、业务状态流转、业务 action、document/root 样式副作用、页面私有样式和组件私有样式。
- 组件内只是简单派生数据或简单事件方法时，直接内联实现。
- 组件内出现复用逻辑、局部流程或多个相关临时状态时，抽成组件附近的 useHook。
- 复杂业务数据、长流程异步、订阅推送、流式返回和多 action 协作进入 zustand-store-styleskill「前端仓库」。
- 组件私有状态只保存纯 UI 临时态，例如弹窗开关、输入框草稿、hover、focus。
- 百行以内组件只承载渲染、绑定交互和纯样式；复杂状态和方法进入主仓库对应私有方法或父级方法；禁止调用兄弟组件方法，出现兄弟方法依赖时提升状态到父级或主仓库。
- 百行以上组件出现复杂派生形参时，收敛到组件私有 `useHook.ts`；私有 hook 禁止调用兄弟组件 hook，出现兄弟 hook 依赖时提升状态到父级或主仓库。
- 组件私有 hook 的状态来源只能是自身作用域、主仓库对应私有方法或父级方法；禁止越过边界调用兄弟组件私有方法。
- 组件拆分后保持默认导出风格；禁止为了私有组件使用 `export function Xxx` 或 `export const Xxx`。
- 禁止为单调用点组件制造 props 透传；组件需要的数据优先在自身最小作用域读取仓库、hook 或上下文。
- 卡片类 UI 出现两个以上真实重复时，优先抽最小卡片壳或标题区，不按业务实体名预拆一批实体专用 Card；卡片壳只承载通用外观、点击标题展开、hover action 位和 children，参数只保留 title、icon、actions、isActive 等真实差异。
- 同一页面多张卡片只有标题交互不同、内容不同但不复用时，优先拆少数标题组件或标题渲染片段；内容区默认留在消费点作为 children，只有内容本身跨卡片真实复用或包含独立流程时才拆内容组件。
- 单组件私有动作不要为了拆组件变成 props 传递；动作依赖的 hook/ref 应留在消费组件内，或移动到真正消费该动作的组件内。
- 跨组件共享的按钮文案、接口标签、状态提示等显示名称必须在最小共同作用域归一；视图私有文案放视图目录或消费点，业务状态、请求和流式提示文案按 zustand-store-styleskill 放进切片仓库或业务对象。
- React 中优先依赖组件 props、useMemo、useCallback、useAsyncFn、zustand store 等实际调用点推导类型。
- 抽屉、弹窗、面板等同类交互已有项目统一组件时优先使用统一组件；没有统一组件时不要在同一页面临时混用多个等价实现。

## 前端样式作用域

- 不要创建 CSS 文件.
- 默认不写样式；只有用户明确要求、功能布局必需或修复明确视觉问题时才添加样式。
- 路由入口不放视图样式；视图私有样式对象或内联样式放对应路由目录，组件私有样式放组件内，第三方组件默认外观不改。
- 修改 document.body、document.documentElement、#root/rootElement、document.title、全局 className、overflow、background、height、margin 或滚动锁定，都属于 document 副作用；必须放在拥有该副作用的页面组件或 root layout 的 useEffect 中，进入时记录旧值，卸载时恢复旧值。
- 禁止在 index.tsx、routers.tsx、普通子组件、工具函数或模块顶层执行 document 副作用。
- 组件私有样式只留在真实消费组件的最小作用域；单组件消费时直接内联到 JSX 或组件内局部常量；只有两个以上真实组件共享同一组样式不变量时，才允许提升到最近共同目录的私有样式常量。
- 样式采用内联写法，遵守满足要求即可的写法，不猜测方式加多余样式。
- 禁止为了视觉延续、统一观感、显得更好看或个人审美，擅自给第三方组件添加背景、hover、padding、边框、阴影、颜色等样式；除非项目已有设计系统要求或用户明确要求。
- 需要默认组件或等价实现时，保留组件默认外观；只连接必要数据、事件和状态。

## 后端作用域

- 后端路由入口只负责读取请求、校验输入、调用业务对象或 store action、返回响应；复杂业务流程不要堆在 route handler 里。
- 服务端模块按真实对象或真实子包建目录；只有被多个模块共同消费的对象才上提到更高目录。
- 服务端项目按真实业务对象建目录；目录名必须对应源数据、运行时对象或明确业务能力。
- 查询视图、聚合状态、页面入口、router 聚合、进程入口不是业务对象，禁止伪装成对象目录。
- 模块私有 schema、协议字段、派生值和辅助逻辑留在模块目录内；跨 route 共享的业务状态和动作收敛到同目录 store.ts。
- src/index.ts 是进程入口，src/routers.ts 是路由聚合入口；二者不是业务对象目录，不放业务 action、schema、缓存实例或页面专用工具。
- 服务端约定 `src/runtime.ts` 是运行时配置，`src/routers.ts` 是路由汇总，`src/index.ts` 是入口；入口只启动运行时，不直接挂载业务模块、不手写子模块路径、不承载 Hono 业务接口。
- `src/routers.ts` 同时汇总业务 Hono router 和 Vite web 托管 router；业务模块目录的 `index.ts` 默认导出已经带 basePath 的完整 Hono router，routers.ts 统一 `.route("/", xxxRouter)`。
- 对象边界不是 class 形式要求，而是业务边界要求：状态、配置、缓存实例、schema、派生值和维护不变量的动作应收敛在同一个对象边界内。
- 有状态实体优先封装为对象；状态字段、派生字段和维护不变量的方法必须收敛在同一个对象内。
- 对象拥有的输入契约、schema、缓存和派生状态应一起收敛在对象内；禁止把单个对象私有的 schema 散落成文件级变量。
- 禁止用外部过程函数通过传参修改对象状态，例如 `runtimePortSet(runtime.PORT + 1)`；应暴露 `runtime.portNext()`、`runtime.devInit()` 这类表达业务意图的对象方法。
- 调用方不要读取对象内部状态后计算再写回；状态如何变化由对象方法负责，调用方只触发动作。
- 业务语义可恢复的数据可以持久化；进程资源、句柄、连接、队列等运行时实例不持久化，真正代表会话、上下文或记忆的对象按业务语义决定是否复用。
- 是否有记忆由对象边界决定：client 可复用，thread、session、conversation 不复用则无记忆，复用则有记忆。
- 不要把依赖 `this` 的实例方法裸返回；必须用闭包保持调用对象，例如 `prompt => thread.runStreamed(prompt)`。
- 纯数据转换、无状态工具和单点逻辑不要为了面向对象强行造类；只有需要维护状态、不变量或多处行为协作时才使用对象。
- 对象方法命名使用 variable-styleskill。

## 导出边界

- 杜绝无外部调用的 export。
- 页面、路由入口、私有组件文件默认使用 default export；只有跨文件实际共享的类型、schema、store 工厂、明确 API 才使用命名 export。
- 跨文件导出只允许真实外部消费者需要的对象类型、schema、store 工厂或明确 API；私有流程、私有协议类型和私有工具不导出。
- 禁止创建只包含 `export type ... from ...`、`export { ... } from ...` 或单纯转发 default 的文件；除非它是包级 public API 边界且有多个真实外部消费者。
- Hono 模块目录 index.ts 运行值默认导出完整 router；聚合入口可命名导出 `XxxApi = typeof app` 这类 API 类型；store.ts 默认导出切片工厂；私有工具和私有类型不导出。
- package.json 的 `exports` 默认不写；只有包级真实 public API 被包外消费者直接导入时才写。
- 需要 `exports` 时必须使用最短字符串映射：单入口写 `"exports": "./src/index.ts"`，多真实入口才写 `{ ".": "./src/index.ts", "./xxx": "./src/xxx/index.ts" }`；禁止为同一路径写 `{ "types": ..., "default": ... }`、`types/import/default` 条件对象或套娃配置，除非发布构建产物确有不同类型文件和运行文件。
- 禁止把 `exports` 当短路径别名使用；不要导出 `./package.json`、`./routers`、`./store`、`./admin-web`、`./user-web` 这类只为省 import 路径、暴露内部目录或跨应用私有实现的子路径。
- web/app 子项目默认不对外暴露 source exports；跨包需要类型、router 或对象能力时，先确认真实 public API 边界，优先收敛到单一包入口或真实 lib，不为了当前调用点临时开子路径；`*Api` 类型应从服务端包唯一 public API 入口导出，不为它暴露 `./routers`、`./store` 等内部子路径。
- lib 库导出只写最短 `exports`，不要写啰嗦的 `types/import` 配置。
